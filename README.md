# upd1c-chains — калькулятор цепочек обновлений 1С

Парсит `v8cscdsc.lst` (список всех обновлений, который получает конфигуратор
с `downloads.v8.1c.ru`), консолидирует его в граф обновлений в PostgreSQL и
даёт веб-калькулятор: «с версии X на версию Y — какие `.cfu` накатить и в
каком порядке».

Один стек: **Node.js + TypeScript** везде. Один Docker-образ обслуживает и
веб, и фоновый импорт. Без отдельного фронт-контейнера.

## Архитектура

```
fetch-lst.ts      ИТС (Basic auth) или локальный файл  ─┐
lst-parser*.ts    стрим-парсер скобочного формата       ├─> update_edges (Postgres)
version.ts        каноническая версия = первый 4-токен  ─┘
import-lst.ts     хэш-дельта: файл целиком + каждое ребро
chain.ts          рекурсивный CTE: кратчайший путь в графе
server.ts         Fastify: API + статичная веб-морда
worker.ts         миграции + импорт по cron
```

### Ключевые решения (зафиксированы по ходу проектирования)

- **Версия = первый 4-сегментный токен.** `1.3.93.1/3.1.46.23` → `1.3.93.1`.
  Часть после `/` не используется и **не хранится** — канонизация на входе,
  в парсере. Подтверждено двумя рабочими 1С-эталонами.
- **Исправлен латентный баг 1С:** старая `СравнитьВерсии` падала на
  `Число("1/3")` и молча возвращала «несравнимо» для всех отраслёвок —
  цепочки по ним не строились. Здесь сравнение по чистому `core`.
- **Переходы между редакциями не строятся** (1.0 → 2.0 → 3.0) — это
  отдельная процедура, не накат `.cfu`. Фильтр по первому сегменту версии.
- **PostgreSQL, не Mongo:** цепочка — это граф, путь считается рекурсивным
  CTE; сырьё источника лежит в `JSONB` (документная гибкость там, где нужна).
- **Двухуровневая хэш-дельта:** одинаковый файл → импорт целиком no-op;
  иначе upsert только изменённых рёбер.

## Что проверено (на реальном `sample.lst`)

- Парсер — дословный порт рабочего 1С `ПарсерLST`, парность подтверждена
  вручную на сырых данных; стрим-версия даёт **точное совпадение** с
  оракулом (3211/3211 записей, все поля).
- Импорт идемпотентен: повторный прогон тех же данных — 9930 → 9930 рёбер,
  ноль дублей.
- Strict-typecheck чистый по всем модулям.
- Логика всех API-эндпоинтов проверена на той же SQL, что в коде
  (342 конфигурации, 9930 рёбер, поиск, группировка по редакциям,
  cross-edition guard).

## Деплой на чистую Ubuntu

```bash
unzip upd1c-chains.zip && cd upd1c-chains
./deploy.sh
```

`deploy.sh` поставит Docker при отсутствии, создаст `.env` (со случайным
паролем БД), соберёт и поднимет стек. Дальше:

```bash
nano .env          # вписать ITS_LOGIN / ITS_PASSWORD
docker compose restart worker
docker compose logs -f worker   # прогресс импорта
```

Веб-морда: `http://<host>:3000/`

### Режим разработки без ИТС

Положить `v8cscdsc.lst` в `./data/`, в `.env` указать
`LST_FILE=/data/v8cscdsc.lst` — импорт пойдёт из файла, без обращения к ИТС.

## Локально (без Docker)

```bash
npm install
docker run -d -p 5432:5432 -e POSTGRES_PASSWORD=upd -e POSTGRES_USER=upd -e POSTGRES_DB=upd postgres:16
export DATABASE_URL=postgres://upd:upd@localhost:5432/upd
npm run generate && npm run migrate
LST_FILE=./data/v8cscdsc.lst npm run import:lst
npm run server          # http://localhost:3000
```

## Дальше по дорожной карте

релизов 1С). Архитектура под это готова: каждый источник — свой адаптер,
пишущий в те же `update_edges` + `raw_json`; правила слияния полей по
приоритету источника. Это следующий этап.
